# MouseStim v8.2 Power-Reduction Roadmap

Purpose: capture the remaining hardware levers for further idle/chronic current reduction now that this board (v8.1) has reached its software-side floor.

Companion docs:
- `mousestim_current_architecture.md` — measured/derived current budget for v8.1.
- `bgm220s_low_power_diagnosis.md` — firmware-side diagnosis for the BGM220S.

## 1. Where v8.1 stands today

| State | Idle current at battery | Notes |
|---|---:|---|
| Original | ~1.47 mA | Pre-investigation baseline. |
| MCU-side firmware optimizations applied (EM2 sleep, PTI removed, etc.) | ~1.34–1.40 mA | MCU floor reached; remaining draw is +5 V analog. |
| Howland GND-shunt trace cut on ADG1236 outputs | ~870 µA | Eliminated ~530 µA of DC leakage through R5/R12. |
| **Software optimization ceiling on v8.1** | **~870 µA** | Further software wins exist only during the brief pre-sync phase, which is negligible time-weighted in chronic operation. |

For context, MCU + 3.3 V system alone (with +5 V rail removed) measured 149 µA. So ~720 µA of the present budget is attributable to the +5 V analog front-end.

## 2. Why software optimization is finished on this board

This is a chronic stimulator: post-sync, the device runs continuously for the battery's life, with the LT6020 cycled per pulse and everything else on the +5 V rail held at its active-mode quiescent. Any software lever that only saves power in the **pre-sync** window (DAC software power-down, BLE scanner removal, advertising cycle tuning, IADC clock gating, busy-wait elimination, SPIDRV deinit) recovers µA for minutes out of a months-long deployment — under 0.01% of total energy. The implementation cost outweighs the benefit on this board.

The chronic-relevant software optimizations are already in place: MCU EM2 sleep, sleeptimer-driven wakes, LT6020 SHDN cycling per pulse.

## 3. Remaining hardware levers, prioritized

### 3.1 MAX1853 ~SHDN under GPIO control (top priority)

- **Estimated saving:** ~324 µA at battery (always-on cost on v8.1).
- **Mechanism:** MAX1853 typical IQ is ~165 µA on +5 V. Currently ~SHDN is tied (always enabled). Routing it to a GPIO lets firmware enable it only when the LT6020 is enabled (i.e., during the active stim pulse window).
- **Logic levels:** ~SHDN logic HIGH for +2.5 V < VIN < +5.5 V — compatible with driving from the existing SHDN net (LT6020 EN line at ≥1.7 V) **only if** the GPIO drives 5 V-domain logic; safer to drive from a dedicated GPIO referenced to 3.3 V if the MAX1853 ~SHDN input tolerates 3.3 V high (verify on the production datasheet — older MAX1853 figures show VIH near 2 V, often workable from 3.3 V; confirm before commit).
- **Constraint on v8.1:** no exposed/free GPIO available to externally tap. Must be a board re-spin lever.
- **Side effect to consider:** at chronic stim rates of 80–160 Hz, the LT6020 is on for ~30–60% of each period. The MAX1853 has a non-trivial enable-to-output-stable time (charge-pump start-up). It must be enabled in lockstep with — or slightly ahead of — the LT6020 SHDN edge so that -5 V is valid at the start of the pulse. Plan GPIO sequencing accordingly.

### 3.2 LT6020 → low-IQ op-amp swap (secondary)

- **Estimated saving:** ~150–300 µA at battery during stim train, depending on rate.
- **Trade-off:** most low-IQ precision op-amps drop the SHDN pin, which would lose per-pulse cycling. Detailed math in the prior conversation showed that at chronic rates of 80–160 Hz, the LT6020 cycle duty is high enough that an always-on 17 µA/amp part (e.g., OPA2333) still nets a saving over cycled LT6020. Crossover sits near ~50 Hz: below ~50 Hz, cycled LT6020 wins.
- **Candidates worth a parametric search (verify on current datasheets):**

| Part | IQ/amp | GBW | SHDN? | Notes |
|---|---:|---:|---:|---|
| LT6020 (current) | 90 µA | 50 kHz | yes | Reference. |
| OPA2333 | 17 µA | 350 kHz | no | Strong candidate; faster than LT6020. |
| MAX44245 | 22 µA | 1 MHz | no | Faster still, similar IQ. |
| AD8607/AD8617 | ~50 µA | 400 kHz | varies | Verify SHDN on the specific variant. |
| LT1495 | 1.5 µA | 10 kHz | yes | **Too slow for 90 µs pulse edges**, listed for reference only. |

- **Alternative if cycling matters across all rates:** add a small load switch (e.g., TPS22918, SiP32431) on the op-amp ±5 V supplies, gated by GPIO. Decouples op-amp choice from SHDN availability.

### 3.3 MAX17220 light-load efficiency tweaks (tertiary)

- **Estimated saving:** ~30–80 µA at battery, depending on operating point.
- **Mechanism:** the MAX17220 boost has reduced efficiency at sub-mA loads, which compounds against every µA on the +5 V rail. Pick a successor part optimized for sub-mA Iq, or move the MCU/3.3 V branch off the boost so the boost only powers the analog rail (and can shut down between trains if combined with §3.1).

### 3.4 Reference / switch refresh (marginal)

| Lever | Saving | Notes |
|---|---:|---|
| LT6656 → lower-IQ reference | ~5–10 µA | Verify drift/temperature spec equivalence. |
| ADG1236 → lower-IQ analog switch | ~3–5 µA | Marginal. |
| Magnet sensor power gating | ~few µA | Only relevant if magnet sensor stays unpowered between detections. |

## 4. Architecture considerations to keep in mind

1. **GPIO budget.** v8.1 had no free, exposed GPIO. v8.2 must reserve at least one for MAX1853 ~SHDN; ideally a second for an op-amp / load-switch enable if the SHDN-less op-amp path is not chosen.
2. **MCU rail isolation.** Today the MCU is fed by TPS709 LDO from the +5 V rail. Splitting the MCU/3.3 V branch directly off the battery (or a dedicated low-Iq buck/LDO) lets the +5 V rail be fully gated when not stimulating. This is the largest single architectural lever for lowering pre-sync current on v8.2 if it's a relevant scenario for the device's deployment.
3. **Howland leakage path.** v8.2 should redesign the ADG1236 fail-safe so it does not require shunting Howland outputs through resistors that have a DC path to GND. Either: (a) the cut-trace approach we used in v8.1 hardware-mod, or (b) reroute the Howland inputs so the shunt path doesn't traverse R5/R12 from VDDA.

## 5. Estimated v8.2 idle-current floor

| Configuration | Idle current at battery (estimate) |
|---|---:|
| v8.1 today (post Howland trace cut) | ~870 µA |
| v8.2 with §3.1 only (MAX1853 SHDN GPIO) | ~330 µA |
| v8.2 with §3.1 + §3.2 (op-amp swap) | ~250 µA |
| v8.2 with §3.1 + §3.2 + §3.3 (boost refresh) | ~200 µA |
| v8.2 with full architecture refresh (MCU rail isolation) | ~150 µA pre-sync, ~250 µA chronic |

These are coarse estimates; bench-validate after each lever lands.

## 6. What is explicitly out of scope for v8.2

- DAC software power-down (chronic-mode irrelevant; see §2).
- LFRCO instead of LFXO (timing accuracy regression for stim).
- EM4 deep sleep (incompatible with sleeptimer-driven stim).
- Stim-parameter compression (settle window, charge-balance ratio): clinical decisions, not power optimizations.
